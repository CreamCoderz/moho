Moho is a framework for 3rd party call control and media control applications. It builds upon SIP Servlet (JSR 289) and Java Media Control (JSR 309) APIs by providing an unified model for both call and media control while still exposing direct access the lower JSR 289/309 APIs.

However, the object model is generic enough to allow implementation on top of other protocol, such as Jingle, and other type of communication, such as Instant Messaging.

Understanding call control model in Moho

There are different models for calls and media control. Moho is designed for calls and media are controlled by a 3rd party server application, such as PBX, IVR, Conferencing, and Call Center applications. Moho assumes all the calls have at least their signals controlled by Moho applications. In most cases, media should be controlled by Moho applications as well.

Understanding media control model in Moho

Moho assumes each {@link com.voxeo.moho.Call Call} has media capabilities as long as its media streams are connected to a JSR 309 compliant media server. Once a {@link com.voxeo.moho.Call Call} is in the {@link com.voxeo.moho.Call.State#INPROGRESS INPROGRESS} or {@link com.voxeo.moho.Call.State#CONNECTED CONNECTED} state, {@link com.voxeo.moho.Call#getMediaService(boolean) getMediaService()} can be used to access the media services. In case the media streams are connected in {@link com.voxeo.moho.Participant.JoinType#DIRECT DIRECT} mode, Moho will try to reinvite the media streams back to the media server into {@link com.voxeo.moho.Participant.JoinType#BRIDGE BRIDGE} mode.

The MediaService.input(...) provides control on both DTMF and speech recognition. MediaService.output(...) provides control on both TTS and media playback. MediaService.prompt(...) allows the application to control both outputs and inputs at the same time. MediaService#record(...) allows the application to record both inputs and outputs.

Borrowing the concept from CCXML and JSR 309, Moho uses various join method to connect different call legs.

A {@link com.voxeo.moho.Participant Participant} can join with other {@link com.voxeo.moho.Participant Participant}s.

Individual streams in {@link com.voxeo.moho.MultiStreamParticipant MultiStreamParticipant} can be joined by using JSR 309 {@link javax.media.mscontrol.join.Joinable Joinable} construt.

Moho also supports multiple joins with automatic join degradation.

Understanding the object model in Moho

The key concepts in Moho are

  • {@link com.voxeo.moho.Application Application} defines lifecycle callbacks and inbound call handling for a Moho application.
  • {@link com.voxeo.moho.Call Call} defines a call leg is {@link com.voxeo.moho.event.EventSource EventSource} and {@link javax.media.mscontrol.join.JoinableContainer}.
  • {@link com.voxeo.moho.event.SignalEvent SignalEvent} and {@link com.voxeo.moho.event.MediaEvent MediaEvent} defines the events for both signal and media.
  • {@link com.voxeo.moho.MediaService MediaService} defines all the media functions available to a {@link com.voxeo.moho.Call Call}
  • {@link com.voxeo.moho.conference.Conference} provides all the conference related functions.

Typically an inbound call results in an {@link com.voxeo.moho.event.InviteEvent InviteEvent} sent to the {@link com.voxeo.moho.Application Application}. The application can decide to {@link com.voxeo.moho.event.InviteEvent#accept(Observer... observers) accept}, {@link com.voxeo.moho.event.InviteEvent#reject(com.voxeo.moho.event.SignalEvent.Reason) reject}, or {@link com.voxeo.moho.event.InviteEvent#redirect(com.voxeo.moho.Endpoint) redirect} the {@link com.voxeo.moho.event.InviteEvent InviteEvent}.

Once the {@link com.voxeo.moho.event.InviteEvent InviteEvent} is accepted, a {@link com.voxeo.moho.Call Call} (leg) is formed. {@link com.voxeo.moho.event.Observer Observer} can be added on the {@link com.voxeo.moho.Call Call} to continue monitor and control the leg. The application can further

  • {@link com.voxeo.moho.Call#join() join} the {@link com.voxeo.moho.Call Call} with the media server.
  • {@link com.voxeo.moho.Participant#join(com.voxeo.moho.Participant, com.voxeo.moho.Participant.JoinType, javax.media.mscontrol.join.Joinable.Direction) join} the {@link com.voxeo.moho.Call Call} to another {@link com.voxeo.moho.Endpoint Endpoint}.
  • {@link com.voxeo.moho.Participant#join(com.voxeo.moho.Participant, com.voxeo.moho.Participant.JoinType, javax.media.mscontrol.join.Joinable.Direction) join} the {@link com.voxeo.moho.Call Call} to another {@link com.voxeo.moho.Participant Participant}.

To use the media function on the {@link com.voxeo.moho.Call Call}, simply get the {@link com.voxeo.moho.MediaService MediaService} from the {@link com.voxeo.moho.Call Call}. If the media is not going through the server, the Moho Framework will try to re-invite the media back to the server if possible.

Understanding the programming model in Moho

The Moho programming model is a event-driven model. Moho has a coarse-grained event types to make the application focus on the business logic other than the lower level protocol. It combines with a state-based event dispatching mechanism and one-thread-per-{@link com.voxeo.moho.event.EventSource event source} (in most cases) to make the application much easier to write.

There are two types of EventObject in Moho - SignalEvent and MediaEvent

{@link com.voxeo.moho.event.SignalEvent SignalEvent} represents events from underlying signaling protocol, such as SIP BYE, REFER, INVITE, etc.

{@link com.voxeo.moho.event.MediaEvent MediaEvent} represents the events from progress and status of the execution of media functions.

Call controls can be performed on {@link com.voxeo.moho.event.SignalEvent SignalEvent}, such as {@link com.voxeo.moho.event.InviteEvent#accept(Observer... observers) accept}. Please note that most call control functions are modeled as synchronous methods for simplicity, given the fact that call control functions are finished within relative short time (e.g. max SIP timeout is about 32 seconds).

However, the media control functions are modeled as asynchronous methods because media functions can be arbitrarily long. The result of any media function will be returned as {@link com.voxeo.moho.event.MediaEvent MediaEvents}. If an application wants to wait for a media function to complete before doing other actions. This can be easily achieved by {@link java.util.concurrent.Future#get() Future.get()} since a media function returns a {@link java.util.concurrent.Future Future} to allow call to query its status.

Each {@link com.voxeo.moho.Call Call} is an {@link com.voxeo.moho.event.EventSource Event Source} that can generate both {@link com.voxeo.moho.event.SignalEvent SignalEvent} and {@link com.voxeo.moho.event.MediaEvent MediaEvent}. To get notified, the application has to add an {@link com.voxeo.moho.event.Observer Observer} or an {@link com.voxeo.util.EventListener EventListener} to the {@link com.voxeo.moho.Call Call}.

Event programming usually is associated with state management. The Moho framework supports application-defined state based event dispatching. Application can {@link com.voxeo.moho.event.EventSource#setApplicationState(String) setApplicationState} on each {@link com.voxeo.moho.event.EventSource EventSource}. Concurrent states are also supported by call {@link com.voxeo.moho.event.EventSource#setApplicationState(String) setApplicationState}. The Moho framework will dispatch the event to the appropriate {@link com.voxeo.moho.event.Observer Observer} method based on its {@link com.voxeo.moho.State State} annotation. Please see {@link com.voxeo.moho.event.Observer Observer} for more information.

Note that each {@link com.voxeo.moho.event.EventSource EventSource} mostly likely has system state driven by underlying protocols. But these should be separated from application states. From Moho point of view, application states are simply symbolic names, entirely managed by application.

Understanding the Moho Application Package

Moho Application Package (MAR) is a simply Web ARchive (WAR). A container supports loading MAR should scan MAR to find and load the implementation class of the {@link com.voxeo.moho.Application Application} interface. If multiple implementation classes exist, the implementation class can be designated by MAR's deployment descriptor (WEB-INF/moho.xml). 

  <application name="MohoApplicationName" class="MohoApplicationClassName">
    <parameter name="pName" value="pValue"/>
    ...
  </application>
Otherwise, a random implementation class is loaded.

If a JSR-289 container doesn't support loading MAR directly, Moho applications should be packaged as standard SAR with the following SIP deployment descriptor.

  <servlet>
    <servlet-name>MohoController</servlet-name>
    <display-name>MohoController</display-name>
    <servlet-class>com.voxeo.moho.sip.SIPController</servlet-class>
    <load-on-startup>1</load-on-startup>
    <init-param>
      <param-name>ApplicationClass</param-name>
      <param-value>MohoApplicationClassName</param-value>
    </init-param>
    <init-param>
      <param-name>pName</param-name>
      <param-value>pValue</param-value>
    </init-param>    
    ...
  </servlet>

Understanding the relationship between Moho objects and JSR 289/309 objects

Some of the Moho objects can be mapped into JSR 289 or 309 objects, which allows the application to access the JSR 289/309 API directly.

  • {@link com.voxeo.moho.Call Call} is equivalent of SipSession in JSR 289 and NetworkConnection in JSR 309.
  • {@link com.voxeo.moho.MediaService MediaService} is equivalent of MediaGroup in JSR 309.
  • Mixer is equivalent of Mixer in JSR 309.

In order to prevent lower level access from messing up the states in Moho, the lower level objects has to be accessed via some sort of proxy objects. Additional restrictions might have to be applied.
